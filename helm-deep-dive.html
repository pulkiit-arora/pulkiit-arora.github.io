üéØ What is Helm?

Helm is the package manager for Kubernetes, often referred to as the "apt/yum/brew for Kubernetes." It simplifies the deployment and management of complex applications by packaging them into charts - a collection of files that describe a related set of Kubernetes resources.

$
helm version
version.BuildInfo{Version:"v3.12.0", GitCommit:"..."}

üì¶ Helm Chart Structure

Every Helm chart follows a specific directory structure that makes it easy to understand and maintain. Here's what a typical chart looks like:

Chart.yaml - Chart metadata and dependencies
values.yaml - Default configuration values
templates/ - Kubernetes manifest templates
templates/deployment.yaml
templates/service.yaml
templates/ingress.yaml
charts/ - Sub-charts and dependencies
.helmignore - Files to exclude from packaging

üîß Core Helm Concepts

üìã Charts

Charts are packages of Helm templates that describe a set of related Kubernetes resources. Think of them as the "source code" for your application deployment.

üöÄ Releases

A release is a running instance of a chart in a Kubernetes cluster. When you install a chart, Helm creates a release with a unique name.

üè™ Repositories

Repositories are collections of charts that can be shared and distributed. The official Helm Hub and Artifact Hub are popular public repositories.

üìù Creating Your First Chart

Let's create a simple Helm chart from scratch. This will help you understand the structure and workflow.

$
helm create my-first-chart
Creating my-first-chart

This command creates a new chart with the following key files:

# Chart.yaml apiVersion: v2 name: my-first-chart description: A Helm chart for Kubernetes type: application version: 0.1.0 appVersion: "1.16.0" # values.yaml replicaCount: 1 image: repository: nginx tag: "latest" service: type: ClusterIP port: 80

üîÑ Helm Workflow

1 Chart Development - Create or customize chart templates with Go templating
2 Package Chart - Use `helm package` to create a .tgz archive
3 Install/Upgrade - Deploy to cluster with `helm install/upgrade`
4 Manage Release - Monitor, scale, and maintain your deployment
5 Rollback if Needed - Revert to previous versions with `helm rollback`

üé® Advanced Helm Features

üîß Go Templates

Helm uses Go templates to generate Kubernetes manifests dynamically. This allows for conditional logic, loops, and variable substitution.

{{- if .Values.ingress.enabled -}} apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: {{ include "fullname" . }} spec: rules: {{- range .Values.ingress.hosts }} - host: {{ .host | quote }} http: paths: {{- range .paths }} - path: {{ .path }} pathType: {{ .pathType }} backend: service: name: {{ include "fullname" $ }} port: number: {{ .port }} {{- end }} {{- end }} {{- end }}

üîó Chart Dependencies

Charts can depend on other charts, creating a hierarchy of deployments. This is useful for complex applications with multiple components.

# Chart.yaml dependencies section dependencies: - name: postgresql version: 12.x.x repository: https://charts.bitnami.com/bitnami - name: redis version: 16.x.x repository: https://charts.bitnami.com/bitnami

üí° Helm Best Practices

üîí Use Namespaces - Always deploy charts to specific namespaces for better organization and security.
üìù Version Your Charts - Use semantic versioning for your charts and maintain a changelog.
üîß Externalize Configuration - Use values.yaml and --set flags to make charts configurable.
üß™ Test Your Charts - Use `helm test` and tools like chart-testing to validate your charts.
üìä Monitor Releases - Use `helm status` and `helm get` to monitor your deployments.
